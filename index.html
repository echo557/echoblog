CORS 预请求

允许方法： GET/HEAD/POST
允许Content-Type:
–text/plain
–multipart/form-data
–application/x-www-form-urlencoded
其他限制：
–请求头限制
–XMLHttpRequestUpload对象均没有注册任何时间监听器
–请求中没有使用ReadableStream对象

什么时预请求？
允许使用头
response.writeHead(200,{
‘Access-Control-Allow-Origin’:’*’,
‘Access-Control-Allow-Headers’:’X-Test-Cors’
})

允许使用方法
response.writeHead(200,{
‘Access-Control-Allow-Origin’:’‘,
‘Access-Control-Allow-Headers’:’X-Test-Cors’,
‘Access-Control-Allow-Methods’:’POST,PUT,Delete’
})
允许以下面方式进行跨域访问的最长时间
response.writeHead(200,{
‘Access-Control-Allow-Origin’:’‘,
‘Access-Control-Allow-Headers’:’X-Test-Cors’,
‘Access-Control-Allow-Methods’:’POST,PUT,Delete’,
‘Access-Control-Max-Age’:’1000’
})

HTTP

HTTP-Protocol基础

URI,URL,URN

URI：Uniform Resource Identifier/统一资源标志符
-用来唯一标识互联网上的信息资源
-包括URL和URN

URL：Uniform Resource Locator/统一资源定位器
schema: ftp/http
路由：/path
此类格式都叫URL，比如ftp协议

URN：永久统一资源定位符
在资源移动之后还能被找到

报文

（a） 请求报文
–Method（如Get）+ 路由信息（/path）+ HTTP/1.0(HTTP版本)
–HTTP Header

（b）响应报文
– HTTP/1.0（版本）+status(200) + OK(Code的含义)
– HTTP Header
– HTTP Body（Body 和 Header之间有一个空行）

HTTP方法

用来定义对于资源的操作
常用： GET, POST, PUT, DELETE

HTTP code
–定义服务器对请求的处理结果
–各个区间的CODE有各自的语义
–好的HTTP服务可以通过CODE判断结果
200
401
404
500
503

HTTP

HTTP-Protocol概要

简单例子

输入URL打开网页
AJAX获取数据
img标签加载图片

Cache-Control
一般用法： max-age=100 对静态资源缓存100s
其他方法：

  1. 设置Public,private可以控制在客户端或者代理服务端缓存
  2. must-revalidate 缓存过期后，需要在代理服务端验证通过后才可以继续使用缓存。
  3. no-cache, no-store是否设置使用缓存

缓存验证：

  1. last-modified 配合if-modified-since
  2. etag配合if-none-match

HTTP的header

  1. Content-Type、Content-Encoding等用来约束数据类型
  2. Cookie保持会话信息
  3. CORS实现跨域并保持安全性限制

浏览器输入URL后HTTP请求返回的完整过程：

  1. startTime开始时间>redirectStart(开始跳转)>Redirect跳转>redirectEnd
  2. fetchStart(fetch开始)>App cache
  3. domianLookupStart域名解析开始>DNS开始查找>domianLookupEnd域名解析结束
  4. connectStart开始创建链接>TCP 创建链接(secureConnectStart 开始创建安全链接)>connectEnd创建链接结束
  5. requestStart开始发送请求>Request 发送请求
  6. responseStart开始接收返回>Response接收响应>responseEnd接收返回结束

5层网络模型

物理层： 定义设备如何传输数据
数据链路层：通信的实体间建立数据链路连接
网络层： 为数据在节点之间传输创建逻辑链路
传输层 TCP, UDP：

  1. 为用户提供可靠的端到端（End-to-End）服务
  2. 传输层向高层屏蔽了下层数据通信的细节
    应用层 HTTP,FTP：
  3. 为应用软件提供了很多服务
  4. 构建于TCP 协议之上
  5. 屏蔽网络传输相关细节

HTTP1.1

数据以字符串传输
持久连接
pipeline:同一个连接多个请求
服务端是按顺序返回（串行）
增加host 和一些其他命令：同一台物理服务器可以跑多个web服务（应用在集群中）
占用带宽比较多

HTTP2

数据以二进制传输
同一个连接里面发送多个请求不再需要按照顺序来
头信息压缩（可减少带宽占用）以及推送等提高效率的功能
HTTP2 和CSS，js 发送时并行的

HTTP 三次握手

需要创建TCP connector
TCP连接上可以发送多个请求

  1. HTTP1.0中TCP 在请求创建时创建，等到请求发送响应后，TCP 连接就会关闭
  2. HTTP1.1的连接可以通过声明一直保持
    发送时可以并行
    服务器返回
    三次握手会有开销
  3. HTTP2中TCP连接是可以并发的，对于同一个服务器发起网页连接的时候，只需要一个TCP连接

三次握手时序图
Client(发送SYN=1, Seq=X)>Server
Client<Server(发送SYN=1,ACK=X+1 Seq=Y)
Client(发ACK=Y+1 Seq=Z)>Server

如何估计性能测试并发量

性能业务需求调研

单一需求：一天3W 订单，一天24小时。

问题需要明确：
系统如何产生订单？有几个页面操作？
一个订单产生需要跳转4个页面。

PV：Page View, 页面浏览数
80%的业务在20%时间内完成。比如：80%业务量，3w订单，一个订单需要4个PV，则是12W 业务量，20%时间：4.8h.即 pv/h=120000/4.8=25000. pv/s = 25000/60/60/4=2
那么峰值PV就是 12W/0.8=15W PV
峰值PV/S= 15W/4.8/60/60/4=

复杂需求

  1. 月营业额
  2. 访问订单转化率, 10%
  3. 媒体访问时间： 24h*80%
  4. 生成每个订单平均选购商品数
  5. 订单有效率： 85%
  6. 平均浏览率（IP/PV）
